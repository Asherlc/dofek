# Xcode Cloud

Xcode Cloud is used for automated App Store builds of the iOS app.

## Repo-tracked inputs

The native iOS project lives at `packages/mobile/ios/`. It was originally generated by Expo prebuild and is committed to the repo so cloud builds can use the checked-in Xcode workspace directly.

The main repo-tracked inputs are:
- `packages/mobile/app.json`
- `packages/mobile/ios/Dofek.xcodeproj/project.pbxproj`
- `packages/mobile/ios/Podfile`
- `packages/mobile/ios/Podfile.lock`
- `packages/mobile/ios/Podfile.properties.json`

## Xcode Cloud workflow

The Xcode Cloud workflow itself is configured in App Store Connect, not in this repository. There is currently no checked-in `ci_post_clone.sh` or other Xcode Cloud-specific bootstrap script under `packages/mobile/ios/`.

Keep the App Store Connect workflow aligned with the current repo requirements:
1. Make Node.js and Corepack available.
2. Run `pnpm install --frozen-lockfile` at the repo root.
3. Run `pod install` in `packages/mobile/ios/`.
4. Build the `Dofek` scheme from `packages/mobile/ios/Dofek.xcworkspace`.

## Signing and provisioning

Xcode Cloud handles code signing automatically via the Apple Developer account linked to the development team. Provisioning profiles and certificates are managed in App Store Connect.

## Local development

The native iOS project is checked into git. After modifying JS dependencies or Expo config, regenerate it from the package script:

```bash
cd packages/mobile
pnpm prebuild
```

To install CocoaPods locally:

```bash
cd packages/mobile/ios
pod install
```

## Updating the native project

When adding new Expo modules or changing `app.json`, re-run `pnpm prebuild` in `packages/mobile` and commit the updated native files. After changing pods, run `pod install` and commit the updated `Podfile.lock` so local builds, CI, and Xcode Cloud stay aligned.
