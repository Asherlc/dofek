# Xcode Cloud

Xcode Cloud is used for automated App Store builds of the iOS app.

## How it works

The native iOS project lives at `packages/ios/ios/`. It was originally generated by `expo prebuild` and is now committed to the repo so Xcode Cloud can build it directly.

### Build pipeline

1. Xcode Cloud clones the repo
2. The **post-clone script** (`packages/ios/ios/ci_scripts/ci_post_clone.sh`) runs automatically:
   - Installs Node.js 22 via Homebrew (pre-installed on Xcode Cloud machines)
   - Enables corepack and activates pnpm
   - Runs `pnpm install --frozen-lockfile` at the monorepo root
   - Runs `pod install` to install CocoaPods dependencies (since `Pods/` is gitignored)
3. Xcode builds the `Dofek` scheme from `Dofek.xcworkspace`

### Configuration

- **Bundle ID**: `com.dofek.app`
- **Development Team**: `68LEL35DQ9`
- **Deployment Target**: iOS 15.1
- **JS Engine**: Hermes
- **React Native Architecture**: New Architecture enabled

### Signing and provisioning

Xcode Cloud handles code signing automatically via the Apple Developer account linked to the development team. Provisioning profiles and certificates are managed in App Store Connect.

## Local development

The native iOS project is checked into git. After modifying JS dependencies or Expo config, you may need to regenerate it:

```bash
cd packages/ios
npx expo prebuild --clean
```

To install CocoaPods locally:

```bash
cd packages/ios/ios
pod install
```

## Updating the native project

When adding new Expo modules or changing `app.json`/`app.config.ts`, re-run `expo prebuild` and commit the updated native files. The `Podfile.lock` should also be committed after running `pod install` so that CI uses the same dependency versions.
